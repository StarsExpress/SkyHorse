1872. Merge Stone Game

Stone Game VIII​

Problem 1872 is a DP problem that creates a particularly deep impression on me.

LeetCode's super popular Alice & Bob game.

Alice Always Goes First​

As LeetCode's tradition, Alice moves first just like ladies first.

So let's put ourselves in Alice's shoes and think about her opening move.

The simplest first move is for Alice to merge all stones into one in 1 sweep, bringing the game to an end with only one stone remaining.

Alice earns sum(stones)\text{sum}(stones)sum(stones) points this way.

Bob, having no turn, scores 0 of course.

Alice's sweep gives a score difference of Dn−1=sum(stones)D_{n - 1} = \text{sum}(stones)Dn−1​=sum(stones),

where the subscript indicates this merge included the (n−1)th(n - 1)^{\text{th}}(n−1)th stone.

But What If There Are ≥3\geq 3≥3 Stones?​

Other Options Arise​

With ≥3\geq 3≥3 stones, Alice has more than just that one-and-done sweep.

She can also leave out some stones, letting Bob come up to merge and score.

Suppose Alice chooses to merge only up to the (n−2)th(n - 2)^{\text{th}}(n−2)th stone.

She earns sum(stones[:n−1])\text{sum}(stones[:n - 1])sum(stones[:n−1]) points.

Two stones remain on the table: the rightmost with value stones[n−1]stones[n - 1]stones[n−1],

and the merged one Alice just created with value sum(stones[:n−1])\text{sum}(stones[:n - 1])sum(stones[:n−1]).

Now Bob must merge both (no passing allowed), earning:

sum(stones[:n−1])+stones[n−1]=sum(stones)\text{sum}(stones[:n - 1]) + stones[n - 1] = \text{sum}(stones)sum(stones[:n−1])+stones[n−1]=sum(stones)

The score difference is Dn−2=sum(stones[:n−1])−sum(stones)D_{n - 2} = \text{sum}(stones[:n - 1]) - \text{sum}(stones)Dn−2​=sum(stones[:n−1])−sum(stones).

Notably, Dn−2D_{n - 2}Dn−2​ can be rewritten using Dn−1=sum(stones)D_{n - 1} = \text{sum}(stones)Dn−1​=sum(stones) as:

Dn−2=sum(stones[:n−1])−Dn−1D_{n - 2} = \text{sum}(stones[:n - 1]) - D_{n - 1}Dn−2​=sum(stones[:n−1])−Dn−1​

Should Alice Switch Options?​

Now we have a binary choice: Dn−2D_{n - 2}Dn−2​ vs. Dn−1D_{n - 1}Dn−1​.

(1). If Dn−2>Dn−1D_{n - 2} > D_{n - 1}Dn−2​>Dn−1​, Alice abandons that one-and-done merger,

and chooses to merge only up to (n−2)th(n - 2)^{\text{th}}(n−2)th stone, leaving out (n−1)th(n - 1)^{\text{th}}(n−1)th.

Recall Dn−2=sum(stones[:n−1])−Dn−1D_{n - 2} = \text{sum}(stones[:n - 1]) - D_{n - 1}Dn−2​=sum(stones[:n−1])−Dn−1​

Dn−2>Dn−1D_{n - 2} > D_{n - 1}Dn−2​>Dn−1​ is equivalent to sum(stones[:n−1])−Dn−1>Dn−1\text{sum}(stones[:n - 1]) - D_{n - 1} > D_{n - 1}sum(stones[:n−1])−Dn−1​>Dn−1​

One conversation to describe: "To my dear opponent: I'll give you Dn−1D_{n - 1}Dn−1​.

That's because I have an even stronger option Dn−2D_{n - 2}Dn−2​ ~~ so strong that even after your Dn−1D_{n - 1}Dn−1​ offset, my net differential still exceeds what I'd get if taking Dn−1D_{n - 1}Dn−1​ directly."

(2). Conversely, if Dn−2≤Dn−1D_{n - 2} \leq D_{n - 1}Dn−2​≤Dn−1​, Alice merges everything.

From the same equation: Dn−2≤Dn−1D_{n - 2} \leq D_{n - 1}Dn−2​≤Dn−1​ means sum(stones[:n−1])−Dn−1≤Dn−1\text{sum}(stones[:n - 1]) - D_{n - 1} \leq D_{n - 1}sum(stones[:n−1])−Dn−1​≤Dn−1​

In translation: "Merging only up to (n−2)th(n - 2)^{\text{th}}(n−2)th stone earns me sum(stones[:n−1])\text{sum}(stones[:n - 1])sum(stones[:n−1]),

but my opponent earns Dn−1D_{n - 1}Dn−1​. After offsetting, my net lead is sum(stones[:n−1])−Dn−1\text{sum}(stones[:n - 1]) - D_{n - 1}sum(stones[:n−1])−Dn−1​.

Comparing these two, I find out it's better to also include (n−1)th(n - 1)^{\text{th}}(n−1)th stone in my merger,

taking Dn−1D_{n - 1}Dn−1​ as my net differential."

Because of this reasoning, Dn−2D_{n - 2}Dn−2​ must be assigned the value of Dn−1D_{n - 1}Dn−1​,

reflecting a key insight: when only (n−2)th(n - 2)^{\text{th}}(n−2)th and (n−1)th(n - 1)^{\text{th}}(n−1)th stones remain unmerged,

based on the score comparison, merging both is better than leaving out (n−1)th(n - 1)^{\text{th}}(n−1)th stone.

So the option to stop at (n−2)th(n - 2)^{\text{th}}(n−2)th stone is eliminated. This elimination is due to our problem's own rule: each player plays optimally.

Deriving the Transition from Here​

By the same logic, if there's an option to merge only up to (n−3)th(n - 3)^{\text{th}}(n−3)th stone:

Dn−3=sum(stones[:n−2])−Dn−2D_{n - 3} = \text{sum}(stones[:n - 2]) - D_{n - 2}Dn−3​=sum(stones[:n−2])−Dn−2​

This gives us a binary choice between Dn−3D_{n - 3}Dn−3​ and Dn−2D_{n - 2}Dn−2​.

You've probably spotted the pattern. The transition equation is:

Di=max⁡(Di+1,  sum(stones[:i+1])−Di+1)D_{i} = \max(D_{i + 1}, \; \text{sum}(stones[:i + 1]) - D_{i + 1})Di​=max(Di+1​,sum(stones[:i+1])−Di+1​)

where the index range is 1≤i≤n−21 \leq i \leq n - 21≤i≤n−2. Why this range?

(1). Index 0 is excluded because the problem requires merging at least 2 stones each time.

(2). Index n−1n - 1n−1 is also excluded because it's the base case:

merging all stones at once, covering up to the (n−1)th(n - 1)^{\text{th}}(n−1)th stone.

Base Case: Dn−1=sum(stones)D_{n - 1} = \text{sum}(stones)Dn−1​=sum(stones)

Understanding the Transition​

I once found a description that captures the stories behind quite well:

Given that when only stones i+1,…,n−1i + 1, \ldots, n - 1i+1,…,n−1 remain, the current player will earn a net differential of Di+1D_{i + 1}Di+1​.

If I merge only up to ithi^{th}ith stone and earn sum(stones[:i+1])\text{sum}(stones[:i + 1])sum(stones[:i+1]),

can my earning still beat Di+1D_{i + 1}Di+1​ after being offset?

I. If yes: I'm willing to play a bigger game by seemingly making a sacrifice here.

Give up the known gain in exchange for an even greater upcoming gain.

II. If no: I can't let my opponent take that Di+1D_{i + 1}Di+1​.

Better put it into my pocket now to be my net differential.

You'll also notice our transition here is a bit unusual:

it moves from larger indices to smaller indices.

So when computing, traverse from larger to smaller indices,

rather than using extra DFS and cache which would cost O(n)O(n)O(n) space......

Also, the transition equation tells us each index's state only depends on the immediately previous index, so there's no need to maintain an array costing O(n)O(n)O(n) space.

#include <numeric>
#include <vector>
using namespace std;

int maximizeScoresDifference(vector<int>& stones) // LeetCode Q.1872.
{
    int prefixSum = 0;
    for (const auto& stone : stones)
        prefixSum += stone;

    // Base case: Alice merges everything as the only attempt.
    int maxScoreDiff = prefixSum;

    prefixSum -= stones.back();

    for (int idx = stones.size() - 2; idx >= 1; idx--) {
        if (prefixSum - maxScoreDiff > maxScoreDiff)
            maxScoreDiff = prefixSum - maxScoreDiff;

        prefixSum -= stones[idx];
    }

    return maxScoreDiff;
}

def maximize_scores_difference(stones: list[int]) -> int:  # LeetCode Q.1872.
    prefix_sum = sum(stones)

    # Base case: Alice merges everything as the only attempt.
    max_score_diff = prefix_sum

    prefix_sum -= stones[-1]

    for idx in range(len(stones) - 2, 0, -1):
        if prefix_sum - max_score_diff > max_score_diff:
            max_score_diff = prefix_sum - max_score_diff

        prefix_sum -= stones[idx]

    return max_score_diff
